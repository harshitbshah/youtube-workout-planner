# CLI Reference

Deep-dive guide for the command-line tool (`main.py`). The README covers setup and
basic usage - this covers everything you'd need to understand, customise, or debug it.

---

## Commands

### `--init`
**First-time full scan.** Run this once after cloning and configuring.

1. Creates the SQLite database tables if they don't exist
2. Fetches the entire back-catalogue of every channel in `config.yaml` (all videos, no date limit)
3. Classifies all fetched videos via the Anthropic Batch API

```bash
set -a && source .env && set +a
python main.py --init
```

Takes 20-40 minutes depending on how many channels you have and how large their libraries are.
Safe to re-run at any time - existing videos and classifications are silently skipped, so only
new channels get processed.

**Required env vars:** `YOUTUBE_API_KEY`, `ANTHROPIC_API_KEY`

---

### `--run`
**Weekly incremental update.** What GitHub Actions runs every Sunday.

1. Scans all channels for videos published in the last **8 days** (not 7 - one day buffer for timezone differences)
2. Classifies any newly found videos
3. Generates the weekly plan
4. Refreshes your YouTube playlist

```bash
set -a && source .env && set +a
python main.py --run
```

**Required env vars:** `YOUTUBE_API_KEY`, `ANTHROPIC_API_KEY`, `YOUTUBE_CLIENT_ID`,
`YOUTUBE_CLIENT_SECRET`, `YOUTUBE_OAUTH_REFRESH_TOKEN`, `YOUTUBE_PLAYLIST_ID`

---

### `--dry-run`
**Plan preview without touching YouTube.** Same as `--run` but skips step 4.

Useful for checking what this week's plan would look like before committing, or for
verifying a new channel's videos are being classified correctly.

```bash
set -a && source .env && set +a
python main.py --dry-run
```

**Required env vars:** `YOUTUBE_API_KEY`, `ANTHROPIC_API_KEY`

---

### `--classify-only`
**Resume interrupted classification.** Skips the channel scan entirely and classifies
whatever unclassified videos are already in the database.

Use this when `--init` completes the scan but classification is interrupted mid-run
(e.g. Anthropic API credits run out, network timeout). The scan results are preserved
in the DB so you don't need to re-fetch anything.

```bash
set -a && source .env && set +a
python main.py --classify-only
```

**Required env vars:** `ANTHROPIC_API_KEY`

## Environment Variables

| Variable | Required for | Description |
|---|---|---|
| `YOUTUBE_API_KEY` | `--init`, `--run`, `--dry-run` | YouTube Data API v3 key (read-only channel scanning) |
| `ANTHROPIC_API_KEY` | `--init`, `--run`, `--dry-run`, `--classify-only` | Claude API key for video classification |
| `YOUTUBE_CLIENT_ID` | `--run` | OAuth client ID for playlist write access |
| `YOUTUBE_CLIENT_SECRET` | `--run` | OAuth client secret |
| `YOUTUBE_OAUTH_REFRESH_TOKEN` | `--run` | OAuth refresh token (generated by `scripts/get_oauth_token.py`) |
| `YOUTUBE_PLAYLIST_ID` | `--run` | YouTube playlist ID to refresh each week |

For local runs, put these in a `.env` file and source with `set -a && source .env && set +a`.
For GitHub Actions, add them as repository secrets.

## Configuration Reference (`config.yaml`)

### `channels`
List of YouTube channels to scan. Use the `@handle` format for the URL.

```yaml
channels:
  - name: "HASfit"               # Display name (used in logs and plan output)
    url: "https://www.youtube.com/@HASfit"
```

You can have as many channels as you like. More channels = richer library = better variety,
but also more Anthropic costs on first scan and more YouTube API quota used per run.

---

### `schedule`
One entry per day of the week. Days with `workout_type: Rest` get no video assigned.

```yaml
schedule:
  monday:
    workout_type: Strength        # HIIT | Strength | Cardio | Mobility | Yoga | Pilates | Dance | Rest
    body_focus: upper             # upper | lower | full | core | any
    duration_min: 30              # minimum video length in minutes (default: 0)
    duration_max: 45              # maximum video length in minutes (default: 60)
    difficulty: any               # beginner | intermediate | advanced | any (default: any)
```

**`workout_type`** must match one of the classifier's output labels exactly (case-insensitive).
`Rest` is a special value - no video is selected and the playlist slot is left empty.

**`body_focus: any`** matches any body focus. Use this when you don't care which muscle group
is targeted, or for workout types like Cardio/Yoga/Dance where body focus is less meaningful.

**`difficulty: any`** matches any difficulty. The classifier assigns beginner/intermediate/advanced
based on what the trainer says in the intro, but many videos leave this implicit - setting `any`
avoids unnecessarily narrowing your pool.

---

### `recency_boost_weeks`
Videos published within this many weeks receive a +100 score bonus, making them
significantly more likely to be selected over older content. Default: `24` (6 months).

```yaml
recency_boost_weeks: 24
```

Lower values (e.g. `8`) keep the plan very current but risk running out of candidates
if your channels don't post frequently. Higher values (e.g. `52`) spread selection
across a full year of content.

---

### `max_channel_repeats`
Maximum number of times any single channel can appear in one week's plan. Default: `2`.

```yaml
max_channel_repeats: 2
```

With 3 channels and 6 training days, setting this to `2` means each channel appears
at most twice per week, forcing variety. Set higher if you have fewer channels.

## Database Schema

The SQLite database (`workout_library.db`) has three tables. Open it with
[DB Browser for SQLite](https://sqlitebrowser.org) to inspect or edit values directly.

### `videos`
One row per YouTube video discovered during scanning.

| Column | Type | Description |
|---|---|---|
| `id` | TEXT (PK) | YouTube video ID (e.g. `dQw4w9WgXcQ`) |
| `channel_id` | TEXT | YouTube channel ID |
| `channel_name` | TEXT | Channel display name (from `config.yaml`) |
| `title` | TEXT | Video title |
| `description` | TEXT | Video description (truncated) |
| `duration_sec` | INTEGER | Video duration in seconds |
| `published_at` | TEXT | ISO 8601 publish date |
| `url` | TEXT | Full YouTube URL |
| `tags` | TEXT | Comma-separated tags |

---

### `classifications`
One row per classified video. A video in `videos` with no corresponding row here
is unclassified and will be picked up by the next `--run` or `--classify-only`.

| Column | Type | Description |
|---|---|---|
| `video_id` | TEXT (PK, FK) | References `videos.id` |
| `workout_type` | TEXT | HIIT, Strength, Cardio, Mobility, Yoga, Pilates, Dance, or Other |
| `body_focus` | TEXT | upper, lower, full, core, or any |
| `difficulty` | TEXT | beginner, intermediate, or advanced |
| `has_warmup` | INTEGER | 1 if video includes a warmup, 0 if not |
| `has_cooldown` | INTEGER | 1 if video includes a cooldown, 0 if not |
| `classified_at` | TEXT | ISO 8601 timestamp of when it was classified |

**Fixing a misclassification:** open the DB in DB Browser, go to the `classifications` table,
find the row by `video_id`, and edit the fields directly. Changes take effect on the next
`--run` or `--dry-run`.

---

### `program_history`
One row per video assigned to a plan day. Used to avoid repeating videos within the
8-week history window.

| Column | Type | Description |
|---|---|---|
| `id` | INTEGER (PK) | Auto-increment |
| `week_start` | TEXT | Monday of the week this plan was generated for (ISO 8601) |
| `video_id` | TEXT (FK) | References `videos.id` |
| `assigned_day` | TEXT | e.g. `monday`, `tuesday` |
| `completed` | INTEGER | Reserved for future use (always 0) |

## How Plan Selection Works

For each active (non-Rest) day in your schedule, the planner runs up to 6 query tiers,
stopping at the first tier that returns at least one candidate:

| Tier | Constraints applied |
|---|---|
| 1 | workout_type + body_focus + duration + difficulty + 8-week history + channel repeat limit |
| 2 | workout_type + body_focus + duration + difficulty + 8-week history (drops channel limit) |
| 3 | workout_type + body_focus + duration + 8-week history (drops difficulty) |
| 4 | workout_type + body_focus + 8-week history (drops duration) |
| 5 | workout_type + body_focus (drops history window - repeats now allowed) |
| 6 | workout_type only (drops body_focus - guaranteed to find something if any videos exist) |

If a tier returns candidates, each is scored:
- **+100** if published within `recency_boost_weeks`
- **+40** if from a channel not yet used this week
- **+0-20** random jitter

The planner then picks randomly from the top 3 scorers (not always the highest score -
the jitter and random selection keep week-to-week variety even when the library is large).

## Cost Breakdown

### YouTube Data API
Free. Default quota is 10,000 units/day. Costs per operation:

| Operation | Units |
|---|---|
| Full channel scan (`--init`) | ~100-200 per channel (varies by size) |
| Incremental weekly scan (`--run`) | ~50-100 per channel |
| Playlist clear + repopulate | ~50-100 depending on plan size |

A typical weekly run with 3 channels uses ~500-800 units total - well within the free quota.
Quota resets at midnight Pacific time.

### Anthropic (Claude Haiku via Batch API)
Batch API is 50% cheaper than standard. Pricing as of early 2026:

| Operation | Approximate cost |
|---|---|
| Full channel init (~75-200 videos/channel) | $0.50-1.50 per channel |
| Weekly incremental (0-10 new videos typical) | $0.00-0.05 |
| `--classify-only` after interrupted init | Same as above, picks up from where it left off |

**Practical totals:** $1-3 to initialise 3 channels, then cents per week ongoing.
A $5 Anthropic credit top-up covers initial setup plus 6-12 months of weekly runs.

> Anthropic API credits are separate from a Claude Pro subscription. Add credits at
> [console.anthropic.com](https://console.anthropic.com) → Plans & Billing.

## Common Operations

### Adding a new channel

1. Add the channel to `config.yaml`
2. Run `--init` locally (safe to re-run - existing videos are skipped)
3. Push to GitHub - the next scheduled run will include the new channel's videos

### Inspecting what's in your library

Open `workout_library.db` in DB Browser and run:
```sql
-- How many videos per channel, and how many are classified
SELECT
    v.channel_name,
    COUNT(v.id) AS total_videos,
    COUNT(c.video_id) AS classified
FROM videos v
LEFT JOIN classifications c ON c.video_id = v.id
GROUP BY v.channel_name;

-- Videos classified as a specific type
SELECT v.title, v.url, c.body_focus, c.difficulty
FROM videos v
JOIN classifications c ON c.video_id = v.id
WHERE c.workout_type = 'Strength' AND c.body_focus = 'upper'
ORDER BY v.published_at DESC;
```

### Fixing a misclassified video

Open `workout_library.db` in DB Browser → Browse Data → `classifications` table.
Find the row (filter by `video_id` or sort by `classified_at`), edit the fields, save.

### Clearing history to allow repeats

If your library is small and you want to allow videos from the last 8 weeks to be
reused, delete rows from `program_history` in DB Browser, or delete the whole table
and let `--run` recreate it empty.

### Forcing a reclassification

Delete the row from `classifications` for the video you want reclassified, then run
`--classify-only`. The video will be resubmitted to Claude on the next batch.
